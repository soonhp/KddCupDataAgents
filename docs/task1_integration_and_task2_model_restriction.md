# Baseline Agent Retry Instruction Integration and Model Restriction Confirmation

## 1. Baseline Agent Integration of `retry_instructions.md`

To ensure that the agent can learn from its own mistakes and correct them on a second attempt, we need to extend the baseline agent’s prompt-building logic so that it includes two additional files when a retry is triggered:

1. **`context/retry_instructions.md`** – a plain‑text file generated after the first run containing a human‑readable summary of the failure.  This file lists issues such as encoding problems, schema mismatches, or semantic errors and suggests how to fix them.
2. **`retry_plan.json`** – a machine‑readable representation of planned repair actions, generated by the repair planner, which the agent can use as a checklist.

The recommended integration steps are as follows:

1. **Detect retry mode**: In `evaluation_runner.py` (or `retry_runner.py`), check whether a retry should be performed by examining `retry_decision.should_retry`.  If true, the `retry_artifacts.retry_task_dir` should be used as the input directory for the second run.
2. **Copy context files**: When preparing the retry task directory, copy or generate `context/retry_instructions.md` and `retry_plan.json` into the task’s `context/` subdirectory.  This ensures that the baseline agent’s prompt loader can discover these files alongside the original context.
3. **Extend prompt construction**: Modify the baseline agent’s prompt generation code (in `data_agent_baseline/run/runner.py` or its template) to look for `context/retry_instructions.md` and append its contents to the system/user messages fed into the model.  The agent should prepend a note such as “The following issues were detected during the previous attempt:” before embedding the retry instructions.
4. **Include `retry_plan.json`**: Pass `retry_plan.json` to the agent as a tool or a hidden parameter so that code generation or SQL plans can be adjusted according to the repair recommendations.  For example, if the plan specifies a `fix_output_contract` action, the agent should rename columns or adjust row counts accordingly.
5. **Logging**: Update the `task.log.json` structure to record whether the prompt was augmented with retry instructions and to save the final merged prompt for debugging.

By implementing these steps, the second‑pass execution becomes aware of the first run’s failure modes and can explicitly address them, leading to higher success rates on complex tasks.

## 2. Confirmation of Model Restrictions in Competition Rules

The official DataAgent‑Bench rules adopt a **“develop freely, evaluate uniformly”** policy for model usage.  During local development, participants may use any language model they choose.  However, during official evaluation, all submissions are run against a single organizer‑deployed model **Qwen 3.5‑35B‑A3B** to ensure fairness.  The rules explicitly state that:

- In the evaluation stage, the evaluation system injects the model service address (`MODEL_API_URL`), API key (`MODEL_API_KEY`), and model name (`MODEL_NAME`), and **participants must not hardcode these values**【286669506279337†L370-L399】.
- The organizers’ Qwen 3.5‑35B‑A3B model is the **only allowed primary task‑solving LLM** during official evaluation.  Running any other LLM inside the container as the main reasoning engine is prohibited【286669506279337†L370-L399】.
- Auxiliary models for preprocessing or retrieval are allowed within hardware limits, provided they are not used as the primary solver【286669506279337†L370-L399】.

Therefore, while it is beneficial to experiment with different models during development, the final submission must rely on the organizer‑provided Qwen 3.5‑35B‑A3B model as its main language model.  Any improvements to routing, repair planning, or retry logic should operate on top of this fixed model during evaluation.
